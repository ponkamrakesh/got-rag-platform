> **A Retrieval-Augmented Generation platform indexing the complete *A Song of Ice & Fire* corpus. Every map, family tree, lineage, prophecy, and paragraph is embedded and retrievable.**

**Live Demo:** Deploy on Vercel (see instructions below)

---

## What This Is

A production-grade RAG system built for the *Game of Thrones* literary universe. Unlike generic PDF chatbots, this platform was engineered with architectural depth:

- **Multi-modal ingestion** — Text, maps, heraldry, family trees, and illustrations are extracted, captioned via GPT-4o-mini Vision, and indexed.
- **POV-aware chunking** — Understands that *A Song of Ice and Fire* uses character-named POV chapters. Chunks respect chapter boundaries to preserve narrative coherence.
- **Recursive semantic chunking** — Splits at paragraphs, then sentences, then words. No sentence is severed mid-clause.
- **Source-cited generation** — Every answer cites book, chapter, and page numbers.
- **Evaluation Dashboard** — Built-in RAGAS-style evaluation with Faithfulness, Relevancy, Context Precision, Context Recall, and latency tracking.

This is not a tutorial project. This is an end-to-end, Vercel-deployable intelligence layer over a complex multi-book corpus.

## Setup

### 1. Prerequisites

- Node.js 18+
- Python 3.10+
- OpenAI API key
- Pinecone account + API key

### 2. Environment Variables

```bash
cp .env.example .env.local
```

Fill in:

```env
OPENAI_API_KEY=sk-...
PINECONE_API_KEY=pcsk_...
PINECONE_INDEX_NAME=got-rag
PINECONE_HOST=          # optional if auto-resolved
```

### 3. Install Dependencies

**Web (Next.js):**
```bash
npm install
```

**Python Ingestion:**
```bash
pip install -r scripts/requirements.txt
```

### 4. Add Your Books

Place your PDFs into `data/pdfs/`:
- `A Game of Thrones.pdf`
- `A Clash of Kings.pdf`
- `A Storm of Swords.pdf`
- `A Feast for Crows.pdf`
- `A Dance with Dragons.pdf`
- `The World of Ice and Fire.pdf`
- ...etc

> **Note:** The parser handles images natively. Maps of Westeros, Essos, Braavos, family trees, and heraldry are automatically extracted into `public/extracted/` and captioned by vision AI so they become searchable.

### 5. Run Ingestion Pipeline

```bash
# One-time Pinecone index setup
python scripts/04_setup_pinecone.py

# 1. Extract all text & images
python scripts/01_parse_pdfs.py

# 2. Caption every map and illustration with GPT-4o-mini Vision
python scripts/02_describe_images.py

# 3. Chunk, embed, and upsert into Pinecone
python scripts/03_chunk_embed_upsert.py
```

**No word is left behind.** The chunking engine uses a recursive splitter that prioritizes paragraph boundaries, then sentence boundaries, then word boundaries — with 300-character overlap — so no context is orphaned at chunk edges.

### 6. Run Evaluation (Optional, Recommended)

```bash
python scripts/run_eval.py
```

This executes the golden eval dataset (`eval/eval_dataset.json`) through the live RAG pipeline and produces `eval/results.json`. Commit this file so the Vercel dashboard can render your metrics.

## How the Chunking Works

The ingestion pipeline does **not** naively split every 1000 characters.

1. **Chapter Detection** — PyMuPDF extracts text blocks. If a standalone ALL-CAPS line matches a known POV character (ARYA, TYRION, DAENERYS, PROLOGUE, etc.), it is recorded as a chapter boundary.
2. **Grouping** — Pages belonging to the same (book, chapter) are concatenated into a single narrative run. This preserves continuity across page breaks.
3. **Recursive Split** — The concatenated text is split using a hierarchy:
   - Prefer `\n\n` (paragraph)
   - Fallback to `\n` (line)
   - Fallback to `. ` (sentence)
   - Final fallback to ` ` (word)
4. **Overlap** — Each chunk shares a 300-character window with the previous chunk, ensuring boundary phrases are never lost.
5. **Metadata** — Every chunk carries `book`, `chapter`, `start_page`, `end_page`, `chunk_index`, and `type` (text vs image caption).

Images (maps, trees, sigils) are extracted as separate PNGs and treated as their own chunks after vision captioning, so searching for *"Where is Vaes Dothrak?"* can retrieve the map chunk.

## Evaluation Methodology

We implement **LLM-as-a-Judge** metrics because classical lexical metrics (BLEU, ROUGE) fail on abstractive RAG answers.

| Metric | Definition | Judge Prompt |
|--------|-----------|--------------|
| **Faithfulness** | Is the answer fully supported by retrieved context? | Rate 0.0–1.0 |
| **Answer Relevancy** | Does the answer address the user's question? | Rate 0.0–1.0 |
| **Context Precision** | What fraction of retrieved sentences are relevant? | Rate 0.0–1.0 |
| **Context Recall** | What fraction of ground-truth info appears in context? | Rate 0.0–1.0 |
| **Latency** | End-to-end retrieval + generation time | Measured in seconds |

The dashboard renders a radar chart of aggregate quality, a bar chart of per-question performance, a latency distribution, and a full evaluation log table.

## Roadmap / Extensions

If you want to push this from "solid resume project" to "standout":

- **Hybrid Search:** Add BM25 keyword retrieval (via `rank_bm25`) and merge with vector scores.
- **Re-ranking:** Cross-encoder re-ranker (e.g., `ms-marco-MiniLM`) on the top-20 retrieved chunks.
- **User Memory:** Session-based conversation history stored in Redis/Vercel KV.
- **Citation Highlighting:** Render the exact sentences in source cards that the LLM used.
- **Multi-modal Answers:** Return image thumbnails directly in the chat stream when a map chunk is retrieved.
- **Cloud Ingestion:** Move the Python pipeline into a GitHub Action that runs on every PDF push.
